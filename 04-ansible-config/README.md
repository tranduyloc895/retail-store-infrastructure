# Jenkins Master & Agent — Configuration Management (Ansible)

## 1. Purpose

This directory contains the Ansible Playbooks and Roles that automate the configuration of the Jenkins Master (Controller) and Jenkins Agent on AWS EC2. The configuration follows the **Idempotency** and **Infrastructure as Code (IaC)** principles.

## 2. Assumptions & Prerequisites

To run these playbooks successfully, the local environment (WSL) and the remote environment (AWS) must meet the following conditions:

* **Base infrastructure:** EC2 instances (Ubuntu) have already been provisioned by Terraform (module `03-jenkins-server`).
* **SSH authentication:** The private key (`.pem`) generated by Terraform has been copied to the local WSL machine with strict permissions (`chmod 400 <file.pem>`).
* **Network security:** SSH access is established through AWS Systems Manager (SSM) Session Manager combined with `ProxyCommand`. Port 22 is never exposed to the internet (`0.0.0.0/0`).
* **Inventory:** The file `inventories/dev/hosts.ini` has been updated with the actual `instance_id` of the target EC2 instances.

## 3. Execution Architecture

The playbook is modular, built from Ansible Roles (`site.yaml` acts as the Orchestrator). It contains two plays:

### Play 1 — Jenkins Master

1. **`common`** — APT cache update and base packages.
2. **`java`** — Install OpenJDK 17 (required by the Jenkins Controller).
3. **`docker`** — Install Docker Engine and add the `ubuntu` / `jenkins` users to the `docker` group.
4. **`jenkins`**
   * Clean up old GPG keys and repository files (scorched-earth cleanup).
   * Download the current signing key (`jenkins.io-2026.key`) to satisfy SecureApt verification.
   * Install the Jenkins package and start the service.
   * **Security (zero-log output):** The `initialAdminPassword` is downloaded directly from the server to the local WSL machine using the `fetch` module. The secret is never printed to `stdout`, preventing cleartext logging (CWE-312).

### Play 2 — Jenkins Agent

1. **`common`** — APT cache update.
2. **`java`** — OpenJDK 17 (required by the Jenkins agent process).
3. **`docker`** — Docker Engine (to build container images).
4. **`awscli`** — AWS CLI v2 (for ECR login).
5. **`jenkins-agent`** — Create the `jenkins` user, install SSH `authorized_keys`, and prepare the working directory.

> **Note:** The `kubectl` role is not included in the Agent play. Under GitOps, the Agent never calls the Kubernetes API — ArgoCD handles that. The role files are kept at `roles/kubectl/` for potential reuse.

## 4. Execution Guide

**Step 1: Verify the SSM tunnel**

Make sure the SSM session is running in the background to forward ports and allow the SSH proxy.

**Step 2: Run the playbook**

From the root of this directory (`04-ansible-config`), apply the playbook:

```bash
ANSIBLE_CONFIG=./ansible.cfg ansible-playbook -i inventories/dev/hosts.ini site.yaml
```

**Step 3: Post-execution and unlocking Jenkins**

1. Ansible fetches the initial-admin-password file to the current directory as `jenkins_initial_admin_password.txt`.
2. **[CRITICAL]** Add this filename to `.gitignore` immediately to prevent a secret leak in the source-control system.
3. Read the file with `cat jenkins_initial_admin_password.txt`.
4. Open a browser at `http://localhost:8080` (through the SSM tunnel) and paste the password to complete the Jenkins initial setup.

## 5. Trade-offs

* **`fetch` vs `slurp` + `debug`** — We chose `fetch` because it pulls the secret file directly to the local machine. We give up the convenience of copying it from the terminal output in exchange for the highest security posture (the secret is never dumped to CI/CD logs).
* **Native APT modules vs raw shell** — We replaced every raw `rm` / `curl` with `apt_repository`, `get_url`, and `file (state: absent)`. The YAML becomes slightly longer, but we gain 100% idempotency and a fully declarative playbook.
